Tutorial

Here you can find a complete tutorial including videos and screencasts to acquaint yourself with use and basic principles of PATHFINDER operation.

At a glance

Utility

Seriously, what is PATHFINDER and what is it for?
It is a computer app aiming to help you to extract new knowledge from the analysis of data collected in ExpoFinder .
What the heck is "extract new knowledge"?
As Bernhard Rieder and Theo Röhle claim in their recent text Digital Methods: Five Challenges (2012), while terms such as "digital humanities" or "cultural analysis" are in fashion there are many alert signals about need of a "computational turn", deeper than a simple quantitative increase in resources to study disciplines such as Art History. Both authors wonders about: if these new digital methods are only no more than just another set of tools in the researcher’s arsenal, how can anyone deal with the most fundamental transformations, necessary to challenge established epistemological paradigms? PATHFINDER tries to help in this dispute. It aims not to be simply a new trick to use computer to help store and organize data; it has been built as a true tool to relief the painful task to perform calculations and approximations so that, from what is known, it were possible to deduce information not noticeable naked eye. It is not about "inventing" anything, but about extracting what is already present in the set of information we have collected. That is why we talk about "generating new knowledge".
Okay, but do I need any special skills to handle PATHFINDER? That is, do I need specific training in statistics, mathematics or computer science?
Not at all. And it is so just because PATHFINDER (or PF, for short) perform the necessary calculations by asking the user only for guidance on the bearing for those calculations. PF does not "do our job", but asks us (figuratively) what we want to find out and acts accordingly.
Can PF only work with dataset from ExpoFinder ?
Although its functional design and conception is flexible and it is designed to be adaptable to data source inherent needs, nowadays the answer is positive: currently PF can use only EF data.
And what will I need to work with PF?
To get the data you must have a valid set of credentials (that is, a username and password) to access ExpoFinder. If you don't have one, request it here.
Do I need to be connected to the Internet?
Actually the vast majority of the work is done by your computer and your browser (Chrome, Firefox, Opera or Safari) and not a remote data server, so it would seem that it is not necessary. However, some tasks require it. For example, downloading the data file from ExpoFinder or on-screen mapping. That is why it is essential to be connected.

Operation

Should I take any cautions before starting to work with PF?
A few. Refresh or reload the page implies lose the work session (it should be restarted). If you performed any operation or "touched" the screen (for example, by pressing a link) the browser will ask if you want to carry on the operation or to cancel it, so be careful about it.

The "back arrow" does not work either [ or something similar] of your browser as you might expect. Pressing it usually leaves the page in which you are browsing, but PF does not use that concept, so it can be found with unexpected behaviors.
Is PF only in Spanish?
No, it is in English too, as you can see ;-). PF detects the predefined language of the browser being used and is the one used by default. But if you want to change it, on footer (see below) the international abbreviation of the alternative language in color red is shown; it means, if the interface is in Spanish, will appear EN (for English), and ES if it is in Spanish. If you want to try, you can click here. It can also be selected from settings menu.

It can be seen in the menu area active on each page, and it offers the possible tasks to be ran.
What does a question mark inside square brackets means?
The small question mark inside square brackets ([?]) either they are a link to open a pot-up window containing a particular kind of help, or they denotes a singular point of information, displaying a "speech bubble" as complement to forementioned thing in a given text. In technical jargon they are usually known as tooltips.

How to…?

Before you start: the use of some abbreviations (ms, s, B, GiB, & hellip;) is striking. For what is this?
PF uses different types of nomenclatures, depending on what it alludes to, but always with the support of the most widely accepted transnational standards. In the case of information measures ( bytes , symbol B, not "b"), it uses both the International Metric System, based on multiples of 10, and ISO/IEC 80000-13, based on a binary count. In the first, one kilobyte, abbreviated "kB", is equivalent to 1,000 bytes (103), while in the second one kilobyte, abbreviated "KiB", is equivalent to 1,024 bytes (210).
What is the first thing to do to work with PF?
First of all, access Data screen. And then, it depends:
  1. If an EF data file IS NOT available…
    1. Make sure you have a legitimate set of credenetials to access ExpoFinder. If you don't have one, request it here.
    2. From the Data page, select Panel → Update data. When pop-up window appears, use your credentials and click Validate.
    3. After a short lapse (usually between 30 and 60 seconds), PF will download an information package in JSON format; it will be stored at local disk as an encrypted file, and PATHFINDER will issue a PIN. You must to preserve it in order to access data later.
    4. SUGGESTION: since PF will request a location and a name for the file, it is a good practice for organizational reasons, and since each downloaded file is a "snapshot" of EF data at any given time, to name it following chronological guidelines. As additional benefit, and if the name is assigned according to YYYY-MM-DD pattern (year with four digits, month with two digits, day with two digits; for example, "2018-07-20" will identify a file downloaded on July 20, 2018) will be easier to locate in a large data folder, sorting the files alphabetically by name.
  2. If an EF data file IS available…
    1. From the Data page, select Panel → Load file. When the dialog box is displayed, select the file you want to use and PF will load it. Even though these figures depend on the microprocessor of your computer, PF will process between 37,000 y 45,000 records per second, so for a volume of 500,000 it will take approximately 11 to 13.5 seconds. Actually it usually takes less.
    2. Before loading it, and as a security measure, PF will ask for your PIN, to be kept when file was downloaded. Write it out and click OK. PF will issue an error message if you make a mistake.
    3. When the process is finished, the Data screen will change to show the new operations set you can run.

The theme of the filter raises frequently. What is that?
Working with an excessively large set of data is often impractical and offers not too relevant results when it comes to generating new knowledge about a given topic. For example, if in PF it is intended to inquire about exhibitions in Spain in a specific period, the presence of a group of records dealing with France can distort more than help our purpose. Therefore, the user is urged to always carry out a previous filter. While the concept originates in maths, in fact, it is easy to understand: it works as a "strainer", so that, of the total primary records, only those matching some particular conditions will be worked on. In the aforementioned example, those exhibitions held in Spain; the rest will not will be taken in accout for any operation.
What areas does the filter affect?
When a filter is established its scope is global; that is, only data will be exported, reports will be produced, statistics will be calculated or networks and graphs will be drawn on the primary records matching the filter conditions.
It is interesting, but how does the filter work?
It works as a "funnel" whereby only records matching the specified conditions will "strain." These conditions apply regardless of the order in which they are written (they are not hierarchical) and will take effect according to their coincidence.
A lot of theory and little practice: give me examples …
Well there they go. But do not forget that the filter will get erroneous results or will not get any if you do not respect the logical rules in its construction.
Perhaps the best way is to start from a small scheme. PF separately selects the primary records matching each condition. Then calculates the intersection of both subsets, and, if inclusions and/or restrictions have been established, adds them to the final result.
But an old Spanish proverb says that "there is not a chariot with only a wheel" Is there a limit on the number of conditions I can use?
Absolutely none. You can put as many as you want. But keep in mind the following rules:
  1. Each condition must be calculated as single before set the filter. In fact, each condition is unique (how many authors are called "Picasso"? How many exhibitions have been held in Malaga?), and they only make sense as a filter by linking them together.
  2. If the possible values for a condition are unknown (for example, if it is not known if there is an author named "Picasso") you can click on in the condition row; as specified, PF will show a list of possibilities.
  3. You will have already realized that the operator "contains" does not take into account uppercase, lowercase or accents. PF ever tries to preserve, as much as possible, the functionality offered by the browser being used and, therefore, the implementation of Javascript (the language with which PF is built) may differ between different browsers. Experimentally it has been verified that both Chrome, Firefox, Opera and Safari realizan correctamente la comparación de caracteres diacríticos. That feature only occurs in the operators contain and does not contain. In others, the comparison is strict.
Don't go so fast. What is that "operator"?
Very simple. In maths, operator is intended as "an application between two vector spaces". But because it seems to be a bit complicated we will make it simpler: a "comparator" between two terms. Daily life uses them abundantly; for example, how many students have failed an exam? Or in other words: how many students have a grade lower than C. PF uses the following operators:
  1. contain or does not contain; the term being compared is part of the field on which the search is performed; indifferent to uppercase and tildes; it does not work as expected (making a sort of database wildcard) with purely numerical values; for example, it would not find "1" in the number "111", although it would find "1" on the date "2018-08-21", treated as a text.
  2. is greater than, is less than, is greater than or equal to, is less than or equal to; as labeled; strict operation (takes into account uppercase and tildes); in the textual value of months (January, February,…) the comparison is alphabetical, so "December" is earlier than "June."
  3. be the same as, do not be the same as; just as previous one, also in terms of ignoring uppercase and tildes; it is an exact comparator ("Pablo Ruiz PIcasso" is not equal than "Pablo Ruiz Picasso").
  4. between; it is a special case; if you write two values separated by slash (/) PF will seek for records between left and right values; widely used in dates (between 2010 y 2015, "2010/2015"); it works with text too, ans comparison is strict.
  5. be null, not null; another special case; indicates whether the field should contain a value or be empty.
And what about "modifier?
Called facet too in PF jargon. Makes mention of a logical part of a field content. For example, in EF, the fields "exhibition: venue place" is build by up to three parts, separated by semicolon (;) and in this order: town, region and country; well, each one of them is a "facet"; so, if you try to search for exhibitions held in the city of Madrid (region has the same name) you must to use town modifier. Some of them are interesting, as decade o century for dates or domain for HTML or RSS URIs.
That filter seems powerful. But it can be a bit long to "rebuild" the conditions of one that I use frequently. Can i keep them?
Of course. Notice that there is a section in the upper area that says "Collection". Once a filter is designed, you can save it on your client computer and take it wherever you want (it is a simple text file in JSON format). You can reload it later.

PF also offers "predesigned" filters downloadable from the server. You can access them from the Panel in the Data menu, Collections section.
And once I filtered what I want… than?
Well, it's up to you. If you want to see what you have selected, use the List submenu of the Data menu. PF will show you a paginated table with all the essential information to access each filtered record. However, if you want to obtain a report, go to the corresponding option of the Panel submenu. Later we explain what the reports consist of. You can also perform analysis tasks in this submenu (or in your individual specifications in the upper part of the screen): scheme, relationships, graphs, networks or maps.
I have the doubt of the search. What is that?
Well, simply a generic filter of a single condition that allows you to search in a specific field or in any field using the "contain" operator. Thus, if you search for "picasso", there will be actors who call themselves that, exhibitions whose title is that or in which a "picasso" participates as an author, or as a curator,… It can be very useful in certain circumstances. Once the result is calculated, the filter must be activated to use it.
What a mess…! It seems to be tangled. How can I clarify?
Keep calm. It has several options:
  1. In the Help menu a pop-up window with general application information options is displayed; specific help is shown with each menu and/or submenu, but the general one remains unchanged. It has the following possibilities:
    1. Menu tree. Shows hierarchically alternatives and operations of menus and submenus.
    2. Frequently asked questions (FAQs)) and corresponding answers. In constant expansion and adjustment according to the needs of the users.
    3. Operative. A flow chart very easy to understand with the basic principles of operation.
    4. How to…? A section dedicated to that of "if I wish to perform X the steps to follow are …".
  2. Documents menu has a complete manual.
  3. As stated, in each individual operation there is a specific help.
In general, we do not believe you should feel unassisted. But if so, you have the contact page to communicate as you prefer with the PF development team. It is our duty and pleasure.
